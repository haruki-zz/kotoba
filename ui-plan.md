# Kotoba UI 重写计划（shadcn UI）

## 1. 文档定位
- 目标：为当前 Electron + React 渲染层制定一份基于 `shadcn/ui` 的 UI 重写计划。
- 范围：本次仅制定计划，不改动业务逻辑与 IPC 契约。
- 依据文档：
  - `memory-bank/design-doc.md`
  - `memory-bank/tech-stack.md`
  - `memory-bank/architecture.md`
  - `memory-bank/progress.md`

## 2. 当前现状
- 当前渲染层主要集中在 `src/renderer/app.tsx` 与 `src/renderer/style.css`。
- 现状规模：
  - `src/renderer/app.tsx` 约 `1250` 行
  - `src/renderer/style.css` 约 `409` 行
- 已落地页面：
  - `活動`
  - `単語帳`
  - `復習`
  - `単語追加`
  - `設定`
- 当前 UI 以手写 CSS 为主，尚未引入 `Tailwind CSS` 与 `shadcn/ui`。
- 当前仓库不存在渲染层组件分层，页面状态、数据加载、视图结构耦合较高。

## 3. 本次重写目标
- 引入 `shadcn/ui`，统一基础控件与交互风格。
- 将当前单体式 `App` 拆分为可维护的页面组件与共享组件。
- 在不破坏现有业务能力的前提下，提升：
  - 一致性
  - 可读性
  - 可维护性
  - 后续扩展速度
- 保持全部用户可见文案继续为日语。
- 保持现有主进程、IPC、数据模型、测试语义不变。

## 4. 非目标
- 本轮不重写主进程服务、仓储、SM-2 算法与 AI Provider。
- 本轮不调整产品信息架构，不新增业务页面。
- 本轮不引入与需求无关的复杂状态管理方案。
- 本轮不为了使用组件库而牺牲当前已验证的交互规则。

## 5. 设计与实现原则
- 先拆结构，再替换样式；避免在单体文件中直接“边改边拼”。
- 先迁移通用骨架与基础控件，再迁移具体业务页面。
- `shadcn/ui` 只负责通用 UI 基元。
- 业务特有展示继续保留自定义实现，例如 `活動` heat map。
- 尽量复用现有状态与 IPC 调用逻辑，避免 UI 重写和行为改动同时发生。
- 每完成一个阶段，都执行最小可回归验证，防止页面全部改完后再集中排错。

## 6. 目标结构
- 建议新增目录：
  - `src/renderer/components/ui`
  - `src/renderer/components/layout`
  - `src/renderer/components/shared`
  - `src/renderer/features/activity`
  - `src/renderer/features/library`
  - `src/renderer/features/review`
  - `src/renderer/features/settings`
  - `src/renderer/features/word_add`
- 建议职责划分：
  - `components/ui`：`shadcn/ui` 基础组件
  - `components/layout`：应用壳、导航、页面容器
  - `components/shared`：状态提示、加载态、确认交互、通用表单片段
  - `features/*`：按页面或业务域组织的容器组件与局部展示组件

## 7. 分阶段实施计划

### 阶段 1：基础设施接入
- 目标：让当前工程具备使用 `shadcn/ui` 的前提。
- 执行动作：
  - 引入 `Tailwind CSS` 及 `shadcn/ui` 所需基础配置。
  - 评估并补齐路径别名配置，便于组件生成与导入。
  - 建立全局设计 token，包括颜色、圆角、间距、边框、字体。
  - 保留现有 `style.css`，但逐步缩减其职责，避免一次性删除。
- 产出：
  - `shadcn/ui` 初始化配置完成。
  - 渲染层具备基础 utility class 能力。
  - 形成统一主题基线。
- 验收：
  - 应用可以正常启动。
  - 现有页面在未迁移前不出现样式级回归。

### 阶段 2：应用壳与通用组件落地
- 目标：先稳定全局外壳，建立后续页面迁移的统一底座。
- 执行动作：
  - 抽离顶部标题区、主导航、页面容器、全局通知区。
  - 使用 `shadcn/ui` 替换通用控件：
    - `Button`
    - `Input`
    - `Textarea`
    - `Card`
    - `Badge`
    - `Tabs` 或等价导航封装
    - `Alert`
    - `Separator`
    - `ScrollArea`
  - 统一空态、加载态、错误态、成功态的视觉表达。
- 产出：
  - `AppShell`
  - `PageHeader`
  - `StatusMessage`
  - `EmptyState`
  - `LoadingState`
- 验收：
  - 五个主页面可在新外壳中切换。
  - 顶层导航、通知、页面容器样式统一。

### 阶段 3：页面按风险从低到高迁移
- 目标：分批替换现有页面，降低回归范围。
- 推荐顺序：
  1. `設定`
  2. `単語追加`
  3. `単語帳`
  4. `復習`
  5. `活動`
- 原因：
  - `設定` 结构最简单，适合作为迁移模板。
  - `単語追加` 能验证表单、按钮、状态提示、保存流程。
  - `単語帳` 涉及列表、编辑态、删除确认，适合在中段处理。
  - `復習` 有卡片与评分操作，交互密度更高。
  - `活動` heat map 最特殊，应最后迁移，避免过早陷入定制样式细节。

### 阶段 4：页面级重构细则
- `設定` 页：
  - 使用 `Card` + 表单控件重组参数编辑区与 API Key 管理区。
  - 将保存状态、删除状态、错误提示统一到共享消息组件。
- `単語追加` 页：
  - 抽离输入区、生成结果区、操作按钮区。
  - 保持草稿自动保存、重新生成、保存成功后清理草稿的现有行为。
- `単語帳` 页：
  - 抽离搜索栏、统计摘要、词条列表、编辑表单。
  - 删除确认优先使用 `AlertDialog` 一类模式。
- `復習` 页：
  - 抽离统计摘要、当前复习卡、评分按钮组。
  - 评分按钮视觉层次要清晰，避免误触。
- `活動` 页：
  - 摘要卡片使用 `Card` 体系统一外观。
  - heat map 本体保留自定义渲染，但与周边布局、标题、说明、滚动容器风格统一。
  - `1-5` 级记忆等级统计改为更清晰的卡片或列表视图。

### 阶段 5：状态与组件边界整理
- 目标：减少 `App` 组件体积，保留清晰的数据流。
- 执行动作：
  - 将页面视图拆成 feature 组件。
  - 将纯展示片段继续下沉为子组件。
  - 仅在必要时抽出 hook；避免为了“看起来高级”而过度抽象。
  - 保持状态提升边界清晰：
    - 全局导航与顶层通知保留在 `App`
    - 页面局部状态放回各自 feature
- 验收：
  - `App` 只保留应用壳、页面切换、少量共享状态。
  - 业务页面具备独立维护能力。

### 阶段 6：旧样式清理与一致性收尾
- 目标：删除不再需要的旧 CSS，避免双轨维护。
- 执行动作：
  - 逐步移除被 `shadcn/ui` 和 feature 组件取代的旧类名。
  - 仅保留：
    - 全局 reset
    - 字体与根变量
    - 少量无法自然表达为 utility class 的特例样式
  - 检查页面间间距、字号、按钮尺寸、边框语义是否一致。
- 验收：
  - `style.css` 缩减为少量全局样式。
  - 不存在大面积“新旧样式混搭”区域。

## 8. 验证计划
- 每阶段至少执行：
  - `pnpm lint`
  - `pnpm typecheck`
  - `pnpm test`
- 页面迁移完成后执行：
  - `pnpm test:e2e`
- 需要人工回归的关键路径：
  - 默认进入 `活動`
  - `単語追加` 草稿恢复
  - AI 生成失败时输入不丢失
  - `単語帳` 搜索、编辑、删除
  - `復習` 评分后队列与统计更新
  - `設定` 保存、删除 API Key、重启后回读
  - `活動` heat map 显示、摘要统计、横向滚动与小窗口布局

## 9. 风险与应对
- 风险：`shadcn/ui` 接入会同时引入样式体系变化。
  - 应对：先接入基础设施，不立即全量替换页面。
- 风险：当前 `App` 状态集中，迁移时容易误改业务逻辑。
  - 应对：先抽视图，再替换控件；优先保持 IPC 调用与状态字段不变。
- 风险：`活動` 页是高度定制区域，直接套组件库收益有限。
  - 应对：只统一外围容器与信息卡，heat map 本体继续自定义。
- 风险：旧 CSS 与新样式共存阶段可能出现覆盖冲突。
  - 应对：明确清理顺序，每迁移一块就删除对应旧类。
- 风险：Electron 渲染层引入新依赖后可能影响构建。
  - 应对：每阶段都执行 `pnpm build` 做一次生产构建验证。

## 10. 完成定义
- `src/renderer/app.tsx` 不再承担所有页面细节渲染。
- 页面主要基础控件迁移到 `shadcn/ui` 体系。
- 旧 `style.css` 仅保留少量全局样式与必要特例。
- 现有业务能力、文案语言、默认主界面与测试语义保持一致。
- `pnpm lint`、`pnpm typecheck`、`pnpm test`、`pnpm test:e2e` 全部通过。

## 11. 建议执行顺序总结
1. 接入 `Tailwind CSS` 与 `shadcn/ui` 基础设施。
2. 重做应用壳与通用控件。
3. 先迁移 `設定`，再迁移表单型页面。
4. 处理中等复杂度的 `単語帳` 与 `復習`。
5. 最后迁移 `活動` 与 heat map 周边布局。
6. 清理旧 CSS，跑完整回归。
