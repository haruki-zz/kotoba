# Kotoba 实施计划（仅计划，不含代码）

## 1. 计划说明
- 计划依据：`/Users/haruki/workspace/kotoba/memory-bank/design-doc.md`（v0.3）与 `/Users/haruki/workspace/kotoba/memory-bank/tech-stack.md`。
- 目标范围：实现桌面端 MVP（Windows/macOS），覆盖“单词生成、词库管理、SM-2 复习、设置管理、数据安全与恢复”。
- 约束前提：Electron + React + TypeScript + JSON 文件存储；默认 Gemini Provider；全日语 UI 文案。
- 本文件用途：指导后续开发执行顺序、交付物与验收口径，减少返工。

## 2. 里程碑与节奏
- M1（1-2 周）：可用 MVP（核心流程可跑通）。
- M2（1 周）：稳定性增强（重试、日志、备份恢复、迁移）。
- M3（1 周）：体验优化（导入导出、快捷键、Provider 扩展接口完善）。

## 3. 分步实施计划

### 步骤 1：启动与基线冻结
目标：冻结开发基线，避免环境与版本漂移。
执行指令：
1. 锁定运行时与依赖大版本：Node 22 LTS、Electron 40.x、TypeScript 5.x、React 19.x、Vite 7.x。
2. 建立项目约束清单，明确必须遵守的默认值（模型、超时、重试、判重规则、SM-2 顺序、review_logs 保留策略）。
3. 明确目录结构与命名规范（主进程、预加载、渲染层、服务层、仓储层、测试目录）。
交付物：环境基线文档、目录结构草案、约束清单。
完成判定：团队成员在同一环境可启动空壳工程，约束清单无冲突项。

### 步骤 2：工程脚手架与安全基线
目标：建立可持续开发的工程壳，并一次性落实 Electron 安全底线。
执行指令：
1. 搭建 Electron + React + TypeScript + Vite 工程。
2. 主进程启用 `contextIsolation: true`、`sandbox: true`、`nodeIntegration: false`。
3. 设计 IPC 白名单，定义每个通道的输入输出结构与参数校验规则。
4. 引入 ESLint + Prettier + Husky + lint-staged，建立提交前质量门禁。
交付物：可运行的桌面壳、基础页面路由、安全配置与 IPC 协议文档。
完成判定：应用可在本机打开空白 UI；非法 IPC 调用被拒绝并可观测。

### 步骤 3：领域模型与 Schema 定义
目标：把业务规则固化为类型与 schema，避免后续逻辑分叉。
执行指令：
1. 定义核心实体：Word、ReviewState、ReviewLog、LibraryRoot、Settings。
2. 用 zod 定义 JSON schema，并覆盖字段约束（含 AI 输出长度上限）。
3. 明确 `schema_version=1` 初始结构及后续迁移接口约定。
4. 固化判重与搜索归一化函数规范：`trim + NFKC`、拉丁小写、假名不敏感。
交付物：类型定义说明、schema 文档、归一化规则说明。
完成判定：所有核心数据结构均有唯一 schema 定义与示例。

### 步骤 4：本地存储与可靠写入链路
目标：实现“可恢复、不易损坏、可迁移”的 JSON 持久化基础设施。
执行指令：
1. 实现仓储层：加载、保存、查询、更新、删除。
2. 写入采用“临时文件 + 原子替换 + 串行队列”，禁止并发覆盖。
3. 每日首次写入前生成备份；启动时 schema 校验失败自动回退最近备份。
4. 实现迁移执行器：按 `vN -> vN+1` 顺序迁移，失败即回滚。
交付物：存储流程说明、异常恢复流程说明、迁移策略文档。
完成判定：异常中断后数据可恢复；损坏文件可自动回退并提示。

### 步骤 5：设置模块与密钥管理
目标：完成 Provider 配置管理和 API Key 安全存储。
执行指令：
1. 实现设置数据结构：provider、model、timeout、retries。
2. 默认值固定为：`gemini-2.5-flash`、15 秒、2 次重试、500ms->1500ms 指数退避（含抖动）。
3. API Key 通过 keytar 存系统钥匙串，不写入词库 JSON。
4. 提供设置读写与校验错误提示策略（全部日语文案）。
交付物：设置模块设计稿、密钥存储流程与错误码定义。
完成判定：未配置或无效 API Key 可被准确识别并引导到设置页面。

### 步骤 6：AI Provider 抽象与 Gemini 实现
目标：打通单词生成链路并保证结构化输出可靠。
执行指令：
1. 定义统一 Provider 接口（与 design-doc 对齐）。
2. 实现 Gemini Provider：仅接受单词输入，返回固定四字段 JSON。
3. 增加 AI 调用超时、重试、错误分级处理；仅对可重试错误触发重试。
4. 增加“非日语检测后自动重试一次”与“双层 JSON 校验”。
交付物：Provider 适配设计、Prompt 规范、错误分类表。
完成判定：同一输入可稳定产出可解析 JSON；失败时能给出明确可执行提示。

### 步骤 7：单词新增与草稿机制
目标：完成 MVP 的“输入 -> 生成 -> 预览编辑 -> 保存”闭环。
执行指令：
1. 构建“単語追加”页面：输入、生成、重新生成、结果预览、保存。
2. 草稿默认开启：输入 800ms 防抖保存；切页或关窗前强制保存；保存成功后清理。
3. 保存时执行判重：按标准化 `word` 比较，忽略 `reading_kana` 差异。
4. 重复词弹窗确认“已存在，是否仍保存”。
交付物：新增流程交互稿、草稿状态机说明、异常处理流程。
完成判定：断网或生成失败不丢输入；保存后数据可重启复现。

### 步骤 8：词库管理（列表/搜索/编辑/删除）
目标：建立可维护的词条管理能力。
执行指令：
1. 构建“単語帳”页面：列表展示、搜索、编辑、删除二次确认。
2. 搜索支持单词/读音/解释的包含匹配，并执行统一归一化。
3. 编辑保存走统一仓储写入链路，更新 `updated_at`。
4. 删除后同步更新相关视图与统计。
交付物：词库页面交互说明、搜索行为定义、编辑删除流程图。
完成判定：1 万词条规模下可流畅检索与编辑，无明显卡顿。

### 步骤 9：SM-2 复习引擎与队列
目标：把复习算法与页面行为稳定落地。
执行指令：
1. 实现纯函数 SM-2 引擎，严格按固定顺序：先 EF、再 repetition/interval、最后写时间与成绩。
2. 队列规则固定为 `next_review_at <= now`，含逾期词条。
3. 时间基准使用系统本地时区；“今日完成”按 00:00-23:59 统计。
4. 评分后即时更新 word 的 review_state，并持久化。
交付物：算法说明、边界用例清单、复习状态转移图。
完成判定：同一输入状态与评分得到确定性输出，且与文档公式一致。

### 步骤 10：review_logs 与统计基础
目标：满足 M1 必选日志与 M2 统计扩展基础。
执行指令：
1. 每次评分写入 review_logs（before_state、after_state、grade、reviewed_at）。
2. 实施日志保留上限 50000 条，超限按时间淘汰最旧记录。
3. 预留统计聚合接口（复习次数、正确率、最近连续天数等）。
交付物：日志记录策略、淘汰策略说明、统计口径定义。
完成判定：日志可追溯一次评分前后状态，且容量受控。

### 步骤 11：全日语 UI 与错误体验收敛
目标：保证产品语言一致性与可恢复交互。
执行指令：
1. 全量页面与提示改为日语，包括按钮、标题、弹窗、错误信息。
2. 覆盖关键错误场景：网络失败、超时、API Key 无效、JSON 解析失败、文件损坏回退。
3. 保证错误后用户可继续操作，不被迫重新输入。
交付物：日语文案清单、错误态交互规范、提示语分级规则。
完成判定：抽样检查无中文残留，错误提示可直接指导下一步操作。

### 步骤 12：测试体系（单测 + 端到端）
目标：为核心链路建立自动化回归保障。
执行指令：
1. Vitest：覆盖 SM-2 纯函数、归一化、判重、schema 校验、迁移逻辑。
2. Playwright（Electron）：覆盖新增词条、复习评分、设置保存、异常重试、备份回退关键路径。
3. 建立最小可用测试数据集与跨平台测试矩阵（Windows/macOS）。
交付物：测试计划、用例列表、通过标准。
完成判定：核心回归用例稳定通过，关键失败路径有对应测试。

### 步骤 13：打包发布与验收
目标：产出可安装版本并通过 MVP 验收标准。
执行指令：
1. 使用 Electron Forge 生成 Windows/macOS 安装包。
2. 执行验收清单：启动可用、全日语 UI、四字段生成可保存、词库 CRUD、复习更新、重启不丢数据。
3. 输出发布说明（已知限制、配置要求、问题回滚方式）。
交付物：安装包、验收记录、发布说明。
完成判定：全部 MVP 验收项通过。

### 步骤 14：M2/M3 增强执行
目标：在不破坏 MVP 稳定性的前提下迭代增强。
执行指令：
1. M2：完善日志与可观测性、重试策略参数化、迁移演练与恢复演练。
2. M3：加入 CSV/JSON 导入导出、快捷键、Provider 插件化扩展点。
3. 每项增强先补用例，再灰度发布。
交付物：增强版本需求单、回归报告、灰度与回滚方案。
完成判定：增强功能上线后核心流程回归零阻断。

## 4. 风险清单与应对动作
- AI 输出不稳定：强制 JSON + zod 校验 + 非日语自动重试 + 手动编辑兜底。
- Provider 限流/费用波动：请求节流、失败重试上限、错误分级提示。
- 本地文件损坏：原子写入、每日备份、启动校验、失败回滚。
- 跨平台差异：Windows/macOS 双端构建与关键流程 E2E 回归。
- 需求漂移：默认值以 `memory-bank/design-doc.md` 为单一事实源，变更需同步更新。

## 5. 执行门禁（DoD）
- 每个步骤必须有“交付物 + 完成判定”，未满足不得进入下一步。
- 涉及默认值变更时，必须同步更新 memory-bank 文档并记录变更原因。
- 核心流程（新增、保存、复习）必须在自动化测试中可回归。
- 发布前必须完成数据安全演练：异常中断、文件损坏、迁移失败三类场景。
